02 TRIZ Contradiction RESOLVE, DON'T COMPROMISE

The core fear surfaced: drift → two UI systems sharing only their colors. The user wants ONE whole system, not two UIs. TRIZ frames this as a contradiction to dissolve, not split the difference.

WANT: CONSISTENCY

Shared brand colors, spacing rhythm, fonts & sizes, header style, hover, loaders, labels, modals, buttons, visual effects, some charts. One coherent look & feel — even the home hero drinks the same tokens.

×

WANT: REUSE

Landing = home slightly unique, privacy/about consistent. Gallery consumes MORE of the DS (forms & co). But landing-only visual flourishes are needed nowhere else — no sense hoisting them into the lib.

TRIZ RESOLUTION — THE PLACEMENT RULE

Tokens are universal and enforced → consistency by construction. Lib-membership is PURELY a reuse decision: used in 2+ places → lib; one-off → stays in the app, but still token-driven. Reuse is orthogonal to consistency.

Vertical axis — TOKENS Law of the land. Every pixel of color, spacing, type, rhythm flows from the theme. Consistency is not a review discipline; it is baked into construction.
Horizontal axis — LIB MEMBERSHIP A pure reuse call. Membership answers "how many places?" — never "does it match?". One-off components live in-app, still fed by the same tokens.

Threads that fed the resolution

03 Backcasting FROM FINISHED SYSTEM → FIRST MOVE

Standing at the finished system and walking backward exposed the true build order — and forced a decisive reversal of Axis C.

REVERSAL — AXIS C: for the pet-project stage, DROP lib versioning/publishing. Simplify to direct workspace import (latest-internal only). IDEA · USER

Decisions crystallised while walking backward

  • +Add Storybook to the lib — view components and their states in isolation. Flow: build component+states → verify in Storybook → import into app → use. IDEA · USER
  • ⊘Arbitrary-value ban KEPT — but only bracket/arbitrary values banned (bg-[#hex], mt-[13px]); the full token-mapped Tailwind utility vocabulary stays usable. IDEA · USER
  • ?Open question raised: Astro vs Next for the SSG landing? Next feels like a massive framework for a mostly-static marketing page. IDEA · USER
  • ✱Design-system aesthetic named: BRUTALIST MONO — raw, high-contrast, monospace type, visible structure, hard edges, minimal ornament. IDEA · USER
  • ▶Brutalist-mono REINFORCES the architecture: a tight rigid token scale fits brutalism, so no-arbitrary-value lint becomes the AESTHETIC, not just discipline; a single mono typeface simplifies the theme; minimal-JS aligns with Astro zero-JS islands. Aesthetic and architecture agree. IDEA · COACH
  • 🔒Landing framework LOCKED: ASTRO (SSG, zero-JS islands, React islands consuming shared ui) instead of Next — lighter, homogeneous Vite toolchain with the Vite gallery, fits Brutalist-mono minimal-JS. IDEA · USER

Every later choice collapses into this single spine. The whole session was one idea, discovered from three angles:

Tokens = law A single UI system enforced by tokens; lib-membership is a pure reuse decision, orthogonal to consistency.
Lint = style Brutalist-mono makes the rigid token scale the aesthetic — no-arbitrary-value lint = style, not discipline.
Brutalism = Astro Minimal-JS brutalism justifies Astro zero-JS islands; a homogeneous Astro+Vite toolchain fits the low-friction pet-project goal.
Tokens = build order Theme-first build order flows straight from tokens-as-source-of-truth. Start at the theme, everything else follows.
