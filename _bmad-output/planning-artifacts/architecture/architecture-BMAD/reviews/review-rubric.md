# Architecture Spine Review — Rubric Walker

## Overall verdict

This is a lean, well-targeted spine. All six ADs pass the "one test" — each one names a real divergence risk between independently-built features (Ingest vs. Browse vs. Insights) and states a concrete, mostly-checkable Rule; none read as filler or unnecessary. Version verification is genuinely strong: every Stack entry was checked against the live npm registry during this review and matched exactly (react 19.2.7, vite 8.1.3, astro 7.0.6, turbo 2.10.3, typescript 6.0.3, pnpm 11.10.0, zustand 5.0.14, exifreader 4.41.0, vitest 4.1.10) — this is verified-current, not asserted from memory, and the memlog documents the ExifReader-vs-exifr tradeoff with real evidence. The gaps are concentrated in one place: the spine is airtight about *what happens inside a single Ingest batch* but silent on what happens *across* Ingest actions (repeat "Add more" sessions), which is a real and likely-to-occur divergence point given the UX doc itself flags re-Ingest as unresolved. There's also one bound FR (Storybook, FR-3) with zero architectural surface, one AD (AD-1, package boundaries) whose enforcement is weaker than its siblings, and one operational dimension (monitoring/analytics) that's mentioned in the memlog's reasoning but never makes it into the spine or Deferred.

## Findings

- **high** Repeat-Ingest ("Add more") accumulation semantics are undefined — AD-3 says the Zustand store "holds the full Ingested `Photo[]`" but never states whether a second Ingest action appends to or replaces that array, nor whether the 100-photo cap (AD-2) is per-batch or per-store-total, nor whether re-adding the same file twice (per-session `id`, not filename-derived, per AD-4) is deduped. (ARCHITECTURE-SPINE.md AD-2, AD-3; EXPERIENCE.md flags the "Add more" control itself as an unresolved `[ASSUMPTION]`, so nothing upstream resolves this either.) *Fix:* add a Rule (to AD-3, or a new short AD) stating explicitly: append vs. replace on re-Ingest, whether the 100-cap is per-batch or cumulative, and whether duplicate files across batches are deduped or double-counted in Insights.
- **medium** FR-3 (Storybook-driven component development) is listed in the frontmatter `binds` but has no architectural surface anywhere in the document — absent from the Capability → Architecture Map, absent from `packages/ui` in the Structural Seed, absent from Stack, absent from Deferred. Two contributors adding components to `packages/ui` have nothing in the spine telling them Storybook is mandatory, where stories live, or that it's CI-gated. (ARCHITECTURE-SPINE.md, Capability → Architecture Map table; frontmatter `binds` line 11.) *Fix:* add an FR-3 row to the Capability → Architecture Map (governed by AD-1 or a new lightweight AD), or explicitly move it to Deferred with reasoning if it's genuinely not architecturally load-bearing.
- **medium** `Photo.thumbnailUrl`'s lifecycle crosses the same worker/main-thread boundary AD-2 otherwise owns carefully, but is only specified as a type comment ("client-generated object URL, revoked on Ingest reset") rather than a Rule. It's unstated whether the worker or the main thread calls `URL.createObjectURL`, and "Ingest reset" isn't itself a defined event (see the high finding above — repeat-Ingest semantics are undefined, so its trigger for revocation is undefined too). Two features touching photo lifecycle (Ingest replacing a batch vs. Browse's "Clear filters" vs. tab close) could implement revocation inconsistently, causing either memory leaks or premature revocation of URLs still rendered elsewhere. (ARCHITECTURE-SPINE.md AD-4, `Photo` type definition, line 82.) *Fix:* extend AD-2 (or add a short rule) naming the single place `thumbnailUrl` is created and the single trigger point for revocation.
- **medium** AD-1's Rule is the least enforceable of the six ADs. AD-5 is backed by a CI-gated ESLint rule (`no-arbitrary-value`) and AD-6 by required Vitest coverage, but AD-1 ("dependency direction is strictly `apps/* → packages/ui → packages/theme`... A component used by two or more apps must live in `packages/ui`, never be copy-pasted") names no automated check — no import-boundary lint rule (e.g. `eslint-plugin-boundaries`, dependency-cruiser) is specified in `packages/eslint-config`, so a cross-boundary import or a duplicated component would only be caught by code review. This is the AD most directly responsible for preventing package-boundary divergence, and it's the one with a purely prose-level Rule. (ARCHITECTURE-SPINE.md AD-1, and Consistency Conventions "Lint / CI gate" row which only names `no-arbitrary-value` + Vitest.) *Fix:* name an import-boundary lint rule alongside `no-arbitrary-value` in `packages/eslint-config`, and add it to the CI gate row.
- **medium** Monitoring/analytics is a silently-skipped operational dimension. The memlog's own rationale for choosing Vercel cites "Vercel Analytics + Speed Insights already in use" on the sister repo as a parity argument (`.memlog.md` line 11), but the spine's Stack table only commits to hosting/deploy targets — analytics, error tracking, and monitoring are mentioned nowhere in the spine body or in Deferred, despite the PRD's load-bearing privacy NFR ("no analytics that capture image content") making this exactly the kind of boundary an architecture document should draw. Left silent, one epic could wire up Vercel Analytics/Speed Insights while another doesn't, or an analytics call could be added that inadvertently captures `Photo` field values, with nothing in the spine to catch it. (ARCHITECTURE-SPINE.md Stack table, line 122; contrast with `.memlog.md` line 11.) *Fix:* add an AD or Deferred bullet scoping analytics explicitly, e.g. "Vercel Analytics/Speed Insights, page-view only, both apps; no event payload may include `Photo` field values."
- **low** No AD or Stack entry states that there is deliberately no client-side router. PRD/UX consistently imply it (Gallery's Ingest/Insights/Browse are `UnderlineTabs`-driven local state, not routes; Landing is "single scroll page, no client-side routing" per EXPERIENCE.md) but the architecture spine never says so itself — a builder working from the spine alone (its stated purpose is to be the "build-substrate") could reach for React Router/TanStack Router by default. Low severity because the UX doc already de-risks this for anyone who reads it, but it is a dimension the spine is silent on. (ARCHITECTURE-SPINE.md Stack table; contrast with EXPERIENCE.md line 46.) *Fix:* add a one-line Rule (e.g. under Consistency Conventions "State & cross-cutting") stating navigation is local component/store state only, no routing library in either app.
